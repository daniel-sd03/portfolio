Government API (Integração IBGE)

Visão Geral

Aplicação back-end desenvolvida com Spring Boot que integra dados públicos da API do IBGE, oferecendo uma camada segura, performática e resiliente para consumo dessas informações. O sistema implementa autenticação com JWT, cache com Redis e tratamento padronizado de erros, garantindo acesso controlado e otimização de chamadas externas.

Arquitetura

A aplicação segue uma arquitetura em camadas (Controller → Service → Client → API Externa). O fluxo inclui autenticação via JWT, validação de requisições por filtro de segurança, verificação de cache no Redis e consumo da API do IBGE através de um client dedicado.

Principais Endpoints

A API expõe rotas claras e padronizadas seguindo os princípios REST. Abaixo estão os recursos disponíveis para consumo:


  • GET /states — Retorna a lista de todos os estados brasileiros (UFs).
  • GET /regioes — Retorna as macrorregiões do país (Norte, Sul, etc).
  • GET /states/{uf}/municipalities — Retorna todos os municípios de um estado específico.
  • GET /municipalities — Retorna a lista completa de todos os municípios do Brasil.
  • GET /municipalities/search?name={nome} — Busca e filtra municípios dinamicamente por parte do nome.

Tecnologias Utilizadas

• Java (17+)
• Spring Boot
• Spring Security
• JWT (Auth0)
• Docker & Docker Compose (Infraestrutura)
• Redis (Cache)
• Swagger / OpenAPI (Documentação)
• WebClient
• PostgreSQL
• JUnit 5 / Mockito

Segurança

A autenticação é baseada em JWT, utilizando abordagem stateless. O usuário realiza login e recebe um token que deve ser enviado nas requisições. Um filtro customizado intercepta as requisições, valida o token, recupera o usuário e registra a autenticação no contexto do Spring.

A configuração de segurança define rotas públicas (login e registro), protege endpoints com base em roles e retorna respostas padronizadas para acessos não autorizados.

Integração com IBGE

A comunicação com a API externa é feita através do IbgeClient, utilizando WebClient. O sistema consome dados detalhados do IBGE, abrangendo estados, regiões e municípios. Além de realizar a conversão segura para DTOs da aplicação, a integração permite listagens completas e buscas dinâmicas (como filtrar cidades pelo nome ou por sigla de estado), mantendo toda essa complexidade totalmente desacoplada da regra de negócio.

Documentação

A API foi totalmente documentada utilizando Swagger (Springdoc OpenAPI). A interface interativa do Swagger UI não apenas detalha todos os endpoints, parâmetros e schemas, mas também serve como um mini-portfólio embutido, exibindo a visão geral da arquitetura e as decisões técnicas diretamente na página inicial da documentação.

Cache com Redis

Para otimizar performance, foi implementado cache com Redis utilizando @Cacheable. As respostas da API do IBGE são armazenadas por 1 dia, reduzindo latência e evitando chamadas repetidas ao serviço externo.

Tratamento de Erros

A aplicação utiliza exceções customizadas (ApiException) e um handler global (GlobalExceptionHandler) para padronizar respostas de erro. São tratados cenários como falhas de comunicação, erros da API externa e respostas vazias.

Testes Automatizados

Foram implementados testes unitários e de camada web utilizando JUnit, Mockito e MockMvc. Os testes cobrem cenários de sucesso e erro, garantindo o correto funcionamento dos endpoints, regras de negócio e tratamento de exceções.

Decisões Técnicas

Para garantir um ambiente de desenvolvimento padronizado e fácil de replicar, toda a infraestrutura de banco de dados (PostgreSQL) e cache (Redis) foi conteinerizada utilizando Docker Compose. Graças ao suporte nativo do Spring Boot 3, os contêineres são orquestrados automaticamente na inicialização da aplicação, eliminando a necessidade de configurações manuais complexas.


O uso de JWT permite uma arquitetura stateless, eliminando sessões e facilitando a escalabilidade. A separação entre Client e Service melhora a organização e testabilidade da aplicação. O Redis foi adotado para reduzir dependência da API externa e melhorar performance.


A implementação de um filtro customizado de segurança oferece maior controle sobre autenticação, enquanto o tratamento global de exceções garante respostas consistentes para os consumidores da API.


Uma decisão técnica de destaque foi o uso de Auto-injeção (Self-Injection com @Lazy) para contornar a limitação de chamadas internas (self-invocation) dos proxies do Spring Cache. Isso foi essencial no IbgeService, onde o método de filtro de municípios precisava invocar a listagem completa cacheada sem acionar a API externa novamente e sem causar um loop de dependência circular na inicialização do projeto.

Encurtador de URL Serverless

Visão Geral

Aplicação serverless de encurtamento de URLs desenvolvida com Java 21 e serviços gerenciados da AWS. O sistema permite gerar links curtos e realizar redirecionamento automático utilizando recursos nativos do S3.

Arquitetura

A solução utiliza Amazon S3 como Static Website Hosting para o frontend, API Gateway para expor a API HTTP e AWS Lambda para processar a criação das URLs encurtadas. O redirecionamento final é realizado diretamente pelo S3 através da propriedade WebsiteRedirectLocation.

Tecnologias Utilizadas

• Java 21
• AWS Lambda
• Amazon S3 (Static Website)
• API Gateway (HTTP API)
• Arquitetura Serverless

Decisões Técnicas

O redirecionamento das URLs foi implementado utilizando a propriedade WebsiteRedirectLocation do Amazon S3, permitindo que o próprio S3 execute o redirecionamento HTTP. Dessa forma, evita-se uma nova execução da Lambda no momento do acesso ao link encurtado, reduzindo latência e custo operacional dentro do modelo serverless.


A geração da chave curta é baseada em hash SHA-256 da URL original, garantindo comportamento determinístico (mesma URL gera o mesmo hash) e boa distribuição dos códigos gerados. Como apenas uma parte do hash é utilizada para compor a URL curta, colisões ainda podem ocorrer, sendo tratadas posteriormente no processo de gravação no S3.


Para evitar colisões, foi implementada uma estratégia progressiva no tamanho da chave: o sistema inicia com 3 bytes do hash e, em caso de conflito, aumenta gradualmente o tamanho da chave até um limite máximo. Essa abordagem equilibra legibilidade da URL e garantia de unicidade.


Na criação dos objetos S3 foi utilizado o header If-None-Match no PutObjectRequest, assegurando que objetos já existentes não sejam sobrescritos. O tratamento específico do erro HTTP 412 (Precondition Failed) permite identificar colisões de forma segura, mantendo comportamento idempotente.

Gerador de CPF

Visão Geral

Aplicação web e API desenvolvida com Spring Boot e Thymeleaf para geração e validação de CPFs. O sistema oferece geração aleatória, geração por estado, geração personalizada (informando os 9 primeiros dígitos) e validação/correção automática dos dígitos verificadores. O projeto demonstra competências em desenvolvimento back-end, segurança básica, persistência e integração com templates.

Arquitetura

Aplicação estruturada em camadas (Controller → Service → Repository) usando Spring Boot. A camada de apresentação utiliza templates Thymeleaf (páginas de cadastro, login e geração de CPF). A persistência é feita com JPA/Hibernate e PostgreSQL. A autenticação e controle de acesso são tratados por Spring Security.

Tecnologias Utilizadas

• Java (17+)
• Spring Boot (Web, Security)
• Thymeleaf (templates)
• JPA / Hibernate
• PostgreSQL
• Maven

Endpoints e Páginas

• Páginas Thymeleaf:

/cadastro — página de cadastro de usuários
/login — página de login
/ — página de geração de CPF (formulário principal)


• Endpoints de formulário (POST):

/salvarUsuario — salva novo usuário (após cadastro)
/gerar-cpf-aleatorio — gera ou valida CPF baseado no botão do form (parâmetro "acao")

Decisões Técnicas

A aplicação usa Thymeleaf para renderizar páginas server-side quando a necessidade é de páginas simples de cadastro/login e fluxo rápido de protótipo. Isso facilita integração entre back-end e UI sem criar um front separado.


A lógica de geração/validação do CPF foi isolada em GeradorCpfService, promovendo reutilização e testabilidade. Regras de negócio e validações ficam no Service; Controllers apenas expõem endpoints e templates, seguindo a separação de responsabilidades.


Para autenticação é utilizado Spring Security com configuração central em WebConfigSecurity, garantindo rotas protegidas (p. ex. página de gerar CPF disponível apenas para usuários autenticados) e tratamento básico de sessões.


Usuários são persistidos com JPA através de UsuarioRepository, permitindo futuras melhorias como logs de uso, limite de geração por usuário ou auditoria.
